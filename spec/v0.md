# Gary - Dungeon Master Assistant Agent

## Overview

Gary is an AI assistant designed to help Dungeon Masters for D&D 5e. Being Dungeon Master is a fun and creative enterprise, but can be very lonely and time-consuming. Gary exists to solve this problem: It offers a sounding board to bounce ideas back and forth, and lets DMs create rich, fun adventures without spending time on minutae like inventing names and word-smithing specific descriptions. Gary works _with_, not _for_, the Dungeon Master - it iterates on your ideas, helps you sharpen your vision, and helps you build a world you and your players will find delightful.

## Glossary

- Entity: A D&D object such as an NPC, location, faction, item etc
- Entity description file: A markdown file containing the entity's description
- Summary: A 1-2 sentence description of each entity, found at the top of the entity description file
- Vault: An Obsidian vault containing all notes for the campaign
- Campaign structure: How the user structures their files in a vault
- Campaign.md file: A markdown file at the top of the Vault that gives a high-level description of the campaign

## User Flows

### Flow: User asks Gary for help planning the next session

1. User invokes Gary with a query: "We just wrapped up session 5. Help me plan session 6"
2. Gary begins exploration of the campaign by reading the Campaign.md file
3. Gary lists the campaign structure and finds the session's notes
4. Gary reads the latest session's notes
5. Gary peeks at possibly relevant entity description files and reads files that will actually be useful 
6. Once Gary is finished exploring, it works with the user to help them accomplish their goal

### Flow: Gary asks user questions

1. Gary is brainstorming about some aspect of the campaign, (e.g., a boss's lair)
2. Gary shows the user a UI with several questions, each with a text box for the user and a submit button:
    - "What should be the tone of the lair? Creepy? Grandiose? Something else?"
    - "How does Gargatron keep people away? Guards? Traps? Is the lair just hidden?"
    - (etc)
3. When the user has submitted all the questions, Gary goes on helping them plan

### Flow: Gary makes an edit

1. Gary and the user have decided on the core aspects of the campaign and the user asks Gary to now write the changes
2. Gary creates a TODO list of files that need to be modified
3. Gary shows the user the text it's about to write to the file with two buttons: accept, and reject with comments (next to a text box)
4. If the user instead rejects the edit, Gary tries again, taking the user's feedback into account

### Flow: User compacts conversation
1. Gary and the user have talked for several turns
2. User clicks the compaction button
3. Gary shows a confirmation dialogue
4. User confirms
5. Gary produces a summary of the conversation, preserving:
   - User's initial and current goals
   - Decisions made so far
   - Files modified
   - Next topics
6. Gary continues from where the conversation left off


## Important File Formats

### Vault Structure

Gary does not assume a specific vault structure, but adapts to the user's preferences. However, Gary can assume a _reasonable_ structure exists, such as:

```
CAMPAIGN.md <- this is required
- world/
       - NPCs/
       - Factions/
       - Locations/
- sessions/
       - Plans/
       - Notes/
```

In this example, Gary can assume that all NPCs are found in the NPCs/ directory, and previous session notes are found in sessions/Notes. If Gary wants to create a directory that doesn't exist, it asks the user, for example

Gary: "I see an NPCs directory, but nothing for Player Characters. Should I use world/PCs/ or merge the two?"

### Campaign File

A file at the root of the vault named CAMPAIGN.md or Campaign.md. This is the high-level description of the campaign, containing the user's vision and overall structure.

Gary always reads this file in its entirety when a session starts. Gary also keeps this file up to date with important development, while making sure only high-level information is contained in it - other files can be references using Obsidian's `[[link]]` syntax.

#### Example
```
# The Forgetful King

## Campaign Overview

**Level range:** 1-3
**Duration:** 10-15 sessions
**Themes:** senility, whimsy, quirky

## Core Premise

As time goes on, even great men become... less great. And so, Forthun Greatback invites the party to help him recover his scepter: It's important because there's a ceremony at the end of the month for his granddaughter's wedding, and he can't officiate the union without his scepter. Only, where could it be?

## Key NPCs

- [[Falkon]]: Court wizard
- ...

## Structure

### Act 1

The party finds a "HELP NEEDED" poster outside their inn. They follow the promise of reward

...

### Act 2

...
```

### Entity Description File

All files are simple markdown files that follow Obsidian notation for links (`[[Name of Other File]]`).

When Gary creates a file, it makes sure to always start with a YAML frontmatter containing terse, "machine-readable" key-value pairs:
```
---
Property Name: Value
---
```

This must be followed by a Summary section, containing human-readable prose and `[[Obsidian Links]]`

This summary section helps Gary with progressive disclosure - it can read the header first, and only read the rest of the file if it is relevant. It tells Gary the 3-4 important details it needs to know about the entity.

When Gary opens a file that does not have a Summary, it instead reads the full file and modifies it, adding the summary at the top.

When Gary modifies a file, it always makes sure to reflect the changes in the Summary.

#### Example

location: $VAULT/NPCs/Varen Ashworth.md
```
---
Status: Kidnapped
Location: Unknown
Species: Human
Description: Varen is Lydia Ashworth's 8-year-old son. He was kidnapped by a Goblin raid on Heartwater.
---

## Summary

Human male, 8 years old. Lives in [[Heartwater]] with his family

## Relationships
- [[Lydia Ashworth]]: Mother
- [[Vartan Ashworth]]: Father (dead)

## Details
Full backstory, mannerisms, links to relevant sessions
```

## UI Components

- **Chat pane**: Conversation with the user and Gary
- **Diff viewer**: Shows proposed edits, with accept / reject with comment options
- **Planner**: When Gary needs input, shows structured choices or free text where relevant
- **File browser**: Shows vault structure as a tree, with icons for files skimmed (only read Summary), read, and edited
- **Compaction button**: Shows token count and lets user compact the conversation. 

## Constraints

- Vaults are the source of truth. Gary does not use a separate database.
- Gary never writes to files without user approval
- Gary must read a file before modifying it. For new files, Gary creates them directly.
- Gary's context window is limited: it uses progressive disclosure whenever possible to avoid filling it with unrelated information

## Technical Decisions

- Local first: Modify markdown files in a vault. Do not keep a separate database. Do not assume the vault's structure beyond a CAMPAIGN.md file in the vault root.
- Typescript / electron app. We need something that runs locally and can modify files on disk.
- Clean, well-tested code. Use industry-standard way for linting, running tests, and managing dependencies.
- Use the Anthropic Agents SDK for Typescript
- Progressive disclosure: Gary must always try and peek at a file's summary to not bloat its context window
- Accept/reject: simple "before" and "after" - no inline diff
- Planning mode: simple UI "form" with validation (checks user answered every question)

### Tool Inventory

- `list_directory(path, recursive=False)`: returns file/folder names for exploring vault structure
- `peek(file)`: returns only YAML frontmatter
- `read(file)`: returns full file content
- `write(file, content)`: asks for user permission to write content to file (create or replace content)

## Out of Scope

- Multiple campaign support. Assume the user opens Gary and points it to a $VAULT. No need to "remember" campaigns
- Automatic compaction. Let the user manage it for now.
- Partial file edit. Files will be small, Gary can just repeat whatever hasn't changed
- Support for MCP servers
- Inline diff
- Collaborative editing
- Mobile / cloud
- Voice
